# Liferay Objects Data Schema & Creation Guidelines

This document outlines the best practices, datatype mappings, and naming conventions for creating **Liferay Objects** from a data schema definition. It provides guidance on how to define fields in Liferay DXP/Portal and align them with Headless REST APIs.

---

## 1. Object Configuration Structure

When defining a new Liferay Object, configure the base metadata as follows:
- **Label:** The user-friendly singular name (e.g., `Doctor`).
- **Plural Label:** The user-friendly plural name (e.g., `Doctors`).
- **Object Name (System Name):** The internal camelCase identifier used in database mappings and Java APIs (e.g., `Doctor`).
- **Scope:** Define as `Company` (tenant-wide) or `Site` (scoped to a specific site group).
- **REST Context Path:** The REST endpoint path. Liferay generates this automatically (e.g., `/o/c/doctors`).

---

## 2. System Fields vs. Custom Fields

All Liferay Objects automatically inherit system-managed fields. Do **NOT** create custom fields with these reserved system names:

| System Name | Data Type | businessType | Description |
| :--- | :--- | :--- | :--- |
| `id` | Long | `LongInteger` | Unique primary key |
| `creator` | String | `Text` | User who created the entry |
| `createDate` | Date | `Date` | Timestamp of creation |
| `modifiedDate` | Date | `Date` | Timestamp of last modification |
| `externalReferenceCode` | String | `Text` | External reference identifier (ERC) |
| `status` | Integer | `Text` | Workflow status (e.g., `0` for Approved) |
| `displayDate` | DateTime | `DateTime` | Schedule publish date |
| `expirationDate` | DateTime | `DateTime` | Schedule expiration date |
| `reviewDate` | DateTime | `DateTime` | Expiration review date |

## 3. Data Type & Field Mapping

When translating a logical data schema into Liferay Objects, use the following field type mappings:

| Logical Type | Liferay UI Label | JSON `type` | JSON `businessType` | Example |
| :--- | :--- | :--- | :--- | :--- |
| **Short Text** | Text | `String` | `Text` | `firstName`, `lastName` |
| **Long Text** | Long Text | `String` | `LongText` | `description` |
| **Integer** | Integer | `Integer` | `Integer` | `reviewCount` |
| **Decimal** | Decimal | `Double` | `Decimal` | `rating`, `distanceMiles` |
| **Boolean** | Boolean | `Boolean` | `Boolean` | `inNetwork`, `telehealth` |
| **Date** | Date | `Date` | `Date` | `birthDate` |
| **DateTime** | Date-Time | `DateTime` | `DateTime` | `appointmentTime` |

## 4. Crucial Naming & Casing Rules

Liferay’s automatic camelCase generator follows strict guidelines when turning UI Labels into system names. This directly impacts the JSON keys returned by Liferay’s Headless REST APIs:

### A. The Acronym Rule (URLs, ZIPs, IDs)
Liferay capitalizes all letters of abbreviations and acronyms if they appear in certain positions:
- Label `ZIP Code` becomes system name **`zIPCode`** (capitalized `IP` after lowercased `z`).
- Label `Booking URL` becomes system name **`bookingURL`** (capitalized `URL`).
- Label `Profile URL` becomes system name **`profileURL`** (capitalized `URL`).

> [!WARNING]
> **API Casing Match:** Frontend JavaScript developers calling the Headless API must use the exact casing generated by Liferay (e.g. `zIPCode`, `bookingURL`, `profileURL`) in their fetch response processing. Standard camelCase formats like `zipCode` or `bookingUrl` will cause undefined values.

### B. Standard CamelCase
Generic labels map normally to standard camelCase:
- `First Name` -> `firstName`
- `In-Network` -> `inNetwork`
- `Accepting Patients` -> `acceptingPatients`

## 5. Deployment Options

### Method A: Direct JSON Import (Recommended)
You can import pre-built Object definition JSON files (like [Object_Definition_Doctor.json](file:///Users/wes/Documents/GitHub/liferay-ui-healthcare/liferay-objects/doctor/Object_Definition_Doctor.json)) directly into Liferay:
1. Navigate to **Control Panel** -> **Objects**.
2. Click the **Options** menu (three dots) -> **Import Object Definition**.
3. Upload the `.json` definition file.
4. Click **Publish** on the imported object to activate the database tables and REST API.

### Method B: Manual Creation
1. Navigate to **Control Panel** -> **Objects** -> **Add**.
2. Fill in the Label, Plural Label, and Object Name.
3. Edit the newly created Object and go to the **Fields** tab.
4. Add fields manually, matching the exact **System Name** and **Type** mapped in your schema definition.
5. Click **Publish**.
